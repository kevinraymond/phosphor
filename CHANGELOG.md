## Unreleased

### Added
- **New effect: Cleave — HPSS percussive/harmonic duet (#1798)** — the most-repeated grassroots wish in the trend research ("react to drums separately from melody"), answered literally: two visual voices share the screen, crystalline ice-blue shards stabbing radially outward from a draggable fracture point on every drum hit, threaded through slow warm luminous ribbons that swell and stiffen with melody and pads — a direct visualization of the Fitzgerald harmonic/percussive separation shipped in A14 and consumed by zero effects until now (`percussive_energy`/`harmonic_energy` drive the voices, `harmonic_ratio` crossfades dominance, so a breakbeat is visually shards and a pad bridge is ribbons, shifting exactly as the arrangement does). First **two-voice cohort sim**: one 250K population split by a fixed index hash (~30% shard slots / ~70% ribbon slots, voice id in `flags.z` per the Vessel latch convention — the mostly-dead shard pool IS the burst reservoir, so a long pad section can never convert the population into 6 s ribbons with nothing left to burst). Emission is **hard-partitioned across two atomic counters** so the voices can never starve each other: ribbons claim via `emit_claim()` (`counters[2]`), shards via a second atomic on the previously reserved `counters[3]` (zeroed with the rest each dispatch, read by nothing else — now documented at every counter site), each compared against its own dominance-weighted share of the frame's emit budget; the percussive gate makes burst *timing*, the counter caps burst *size*, and `burst_on_beat` supplies extra budget that inflates harmlessly when no transient fires. Shard bursts gate on `u.percussive_energy` **multiplicatively** (finding #1836: the kick band false-fires ~80% on sustained pads; onset+flux only sharpen the attack and can never open the gate alone) — and the live pad-drone acceptance test refined #1836 itself: the *raw* HPSS percussive value is ~0 on a pure pad, but the shipped uniform is adaptively normalized, and with no drums anywhere in the ~4 s ranging window the normalizer stretches that residue to ~0.7, so percussive level alone still false-fires. The discriminator that survives normalization is `harmonic_ratio` (Passthrough policy, level-invariant): a dominance guard fades strikes out entirely as the mix goes harmonic-pure, verified live (pad drone: kick band pinned high, zero shard bursts, ribbons swelling). The `balance` param biases the same dominance with a full-range swing, so slamming it to either pole is a hard manual override even mid-breakbeat. Both voices ride the Tide ribbon trail renderer: a shard is a short-lived (0.3–0.7 s) ballistic particle whose bright fast-tapering 16-frame trail polyline IS the crystalline needle; ribbons are long-lived (4.5–7.5 s) threads relaxed toward a shared divergence-free curl field in 8 quantized streamline families (coherent strands, never per-particle dither — the Tide lane lesson), stiffening into glass as `harmonic_energy` rises. Both inherit obstacle collision (needles shatter off silhouettes). Background plate crossfades its own feedback advection with the duet — percussive content expands echoes radially (each hit leaves a shockwave ghost), harmonic content weaves them sideways — under cold chromatic decay and a faint key-locked aurora floor, with the mandatory 1.5 HDR clamp doubly needed under two additive voices. Key-locked palettes on both voices (`dominant_chroma` anchor: shards pulled hard toward ice blue-white, ribbons warm with centroid tilt), 8 bindable params including a `balance` override to force either voice by hand. The user live look caught two more depth-charges: the lib's `fract(sin(n)*43758…)` hash degrades on GPU for index-scaled arguments (~10⁶) — a band of particle indices rolls near-constant tiny values that pass ANY spawn threshold, producing an immortal audio-independent starburst — so Cleave's per-index randomness uses an exact integer hash (lowbias32) throughout; and the "coherent ribbon threads" of the first build turned out to be a spawn-clustering artifact of that same broken hash (a divergence-free curl field preserves uniform density forever), so the ribbon voice now earns its filaments honestly with a ridge-drift compressibility term — uniform spawns braid into golden threads that tighten with harmonic content. Guard tests: `cleave.pfx` parse + an ignored GPU probe compiling sim+bg through the production lib concatenation (also validates the `counters[3]` atomic against the shared binding layout).
- **New effect: Vessel — body-as-container fill-and-release (#1797)** — the performer's silhouette (webcam / MiDaS depth / image obstacle) becomes a vessel that slowly fills with trapped light as a musical buildup rises, then bursts outward exactly on the drop, each particle carrying its stored velocity — the most legible structure→visual mapping in the engine, and the first consumer of the A18 buildup/drop detectors in a particle sim. Containment is hand-rolled in the sim (the `Contain`-mode idiom — binary-search boundary placement + inward reflection — built on `obstacle_alpha`/`obstacle_normal`) rather than the canonical collision call, so it works under **any** obstacle panel mode; with no obstacle armed, a centered SDF "amphora" takes the body's place. Fill rate is gated in-shader by `buildup` (an ambient simmer floor keeps silence alive), interior pressure-jitter and per-particle `heat` charge with the riser, and the one-frame `drop` impulse flips a per-particle release latch — the whole trapped population bursts radially in the same dispatch (no global state), then the vessel refills naturally during the detector's 16 s refractory. A `liquidity` param morphs the identity from pooling liquid (gravity accumulation is the fill line, bar-clock slosh) to weightless curl-drift fireflies, and restitution morphs with it (stars ping off the wall, liquid plops). Stranded particles (the silhouette moved off them) get a recapture spring + fast-age drain, the Tide stall pattern. A bindable `release` param (OSC `/phosphor/param/release`) fires the burst by hand for sets without detector-clean drops; held high it turns the vessel into a fountain — intentional. Key-locked palette (`dominant_chroma` anchor, centroid tilt): contained light simmers dim and heats toward white-gold, burst debris flashes near-white and cools to embers. Background plate reads `buildup`/`drop` directly (fragment side has had them since A18): a key-colored charge floor pools upward, feedback drifts gently up while filling and stills at full charge, and the drop shock is injected into the feedback loop so the one-frame impulse decays over the following frames. Guard tests: `vessel.pfx` parse + an ignored GPU probe compiling sim+bg through the production lib concatenation (including the sdf lib the amphora needs — also the pre-launch check that the WGSL uniform mirror matches the 864-byte layout).
- **New effect: Tide — depth-occlusion flow-around-body water (#1796)** — a luminous waterfall pours from the top edge and visibly parts, pools, and eddies around obstacle silhouettes (webcam / MiDaS depth / image): the teamLab "Universe of Water Particles" primitive, built on the shipped collision core (`apply_obstacle_collision` Flow Around) and the per-layer obstacle bindings/OSC from #1793. The laminar look comes from a deterministic shared velocity field (divergence-free curl meander, y-elongated features, 64 quantized lanes) plus exponential velocity relaxation — neighbors ride identical streamlines, so the sheet reads as water, not smoke — and Tide is the **first consumer of the velocity-aligned ribbon trail renderer** (`trail_length: 16`; the renderer shipped with the trail system but zero effects enabled it) and the first real billboard-mode particle effect. HPSS drives the musical contrast end-to-end: `percussive_energy` breaks the sheet into whitewater and upward foam spray (splash is gated on HPSS only — the kick band false-fires on sustained low sines), `harmonic_energy` stiffens relaxation so pads glide as glass, `harmonic_ratio` breathes the meander. Anticipatory obstacle sampling (2 lookahead alpha taps + a normal tap near surfaces) pools water on heads/shoulders, parts streams at the crest with a hash tie-break, and adds signed eddy rotation — all before the canonical collision call, so the look survives any obstacle mode. Background pass is deliberately anisotropic: feedback advected downward (echoes fall with the water), x-only refraction shimmer, chromatic decay that kills red first (older light turns blue-green), harmonic-gated caustic curtains. Key-locked palette (dominant_chroma anchor, centroid warm/cool tilt), 8 bindable params. Verified live: ~214K steady-state particles (250K cap, 300K quality-scaled ceiling) at 100+ FPS with trails on, silhouette parting/pooling/contour confirmed against both a static Image obstacle (OSC-driven) and a live webcam/MiDaS depth check (Flow Around), and the drums-vs-pad A/B confirmed via null-sink synth loops (BPM and key lock picked up the test material). Guard tests: `tide.pfx` parse (discovery silently drops invalid JSON) + an ignored GPU probe compiling sim+bg through the production lib concatenation.
- **HPSS reaches particle sims (#1796 groundwork)** — the A14 harmonic/percussive split (`percussive_energy`, `harmonic_energy`, `harmonic_ratio`) has been computed live since the audio DSP tier landed and was already in the fragment-shader uniforms, but `ParticleUniforms` had no room for it, so no particle effect could react to drums separately from melody. Batched ABI bump (the #1505 pattern): the three fields + one spare pad appended to `ParticleUniforms` (832 → 848 bytes, all existing offsets stable, one WGSL edit in the shared `particle_lib.wgsl` covers all 16 sims) and copied in `update_audio`. First consumer is the upcoming Tide effect; also unblocks Cleave (#1798).
- **Buildup/drop reach particle sims (#1797 groundwork)** — the A18 structure detectors (`buildup`, a smooth 0–1 riser/tension ramp; `drop`, a one-frame trigger latched by the audio thread so no frame can miss it) have been computed and OSC-emitted since the DSP tier landed, and were already in the fragment-shader uniforms — but no particle sim could read them (#1490 dormant-feature finding). Same batched ABI-bump pattern as the HPSS block: the two fields + two spare pads appended to `ParticleUniforms` (848 → 864 bytes, all existing offsets stable, one WGSL edit in the shared `particle_lib.wgsl` covers all sims) and copied in `update_audio`. First consumer is the upcoming Vessel effect (#1797), whose fill-and-burst choreography is driven entirely by buildup → drop.
- **Obstacles join the binding bus and OSC (#1793)** — the obstacle surface was the only major subsystem with zero binding targets and zero OSC addresses: the engine's marquee interactive feature couldn't be automated, cued, or driven by audio at all. Two new control paths. **Bindings** (broadcast to all particle layers, like every other `particle.*` target): `particle.obstacle_enabled` (level-triggered at 0.5), `particle.obstacle_threshold` and `particle.obstacle_elasticity` (0..1 direct), and `particle.obstacle_mode` — a discrete target quantized via a new `ObstacleMode::from_normalized` that spreads the normalized 0..1 bus output evenly across all four modes (0.0 → Bounce, 1.0 → Contain, same pattern as `layer.blend`, #1792), so threshold can breathe on `audio.rms` or mode can hop on a MIDI knob. **OSC** (per-layer, respects layer lock): `/phosphor/layer/{n}/obstacle/{enabled|mode|threshold|elasticity}` with mode as a raw integer 0–3 — so Contain-on-layer-A + Flow-Around-on-layer-B choreography is now cueable from any OSC sequencer, including while the active layer is locked. The new addresses feed the binding-bus source pool and OSC learn like every other typed message. Unlocks the Tide (#1796) and Vessel (#1797) effects. Unit tests cover quantizer endpoints/rounding/clamping, all parse branches (including learn-mode fallbacks for unknown fields), and address round-trips.

### Fixed
- **Tide no longer white-washes on real music (#1796/#1857 fallout)** — four compounding causes, found across four live-look rounds. **(1) Splash gate below the live floor**: the HPSS fix below armed the dormant splash gate (`percussive_energy > 0.6`), tuned against the dead feed and *below* the live sustained-groove floor (~0.63) — the falls converted to foam continuously. **(2) Degenerate hash**: the splash roll used the #1856 fract-sin hash, so a band of particle indices passed every re-roll and became standing white foam streaks. **(3) Uncapped conversion rate**: even with a correct gate band, loud sections park the percussive level above any threshold for seconds, and an uncapped per-particle roll foamed the entire 250K sheet ~3×/s — level alone cannot articulate drum-dense material. **(4) The real white-out term — trail coverage**: total sheet light scales as count × alpha × color × **trail length**, and trail screen-length is proportional to speed; loud music drives the rms-coupled flow ~1.6× faster, multiplying total deposited light by 1.6 with color and alpha held flat — no amount of color/alpha tuning could touch it (three rounds proved this empirically). Fixes: splash gates at `smoothstep(0.72, 0.92)` × an onset-synced attack factor with a small subset coefficient (spray reads as sparks, ≲25% coverage at the hardest hits); splash randomness rolls through a local lowbias32 integer hash; whitewater breakup floored at its designed 0.25 idle (`smoothstep(0.62, 0.95)`); `speed_glow` is compressive (`1−exp`) instead of a linear clamp that saturated at exactly the speeds loud music produces; and alpha is normalized by speed (deposited light per particle is speed-invariant — faster water renders as longer, thinner, dimmer streaks, i.e. motion blur) plus an rms head-room dim, so silence is untouched (compensation ≈1.0 at idle speed) and loud sections land at ~1.0× the idle total instead of 1.6×. Bloom threshold 0.35 → 0.42 (Cleave's proven value). The emit path's #1856 hash idiom is deliberately untouched — its banding is part of the approved look, pending the lib-level audit. Live-verified across silence / quiet / loud DnB; user-approved.
- **HPSS energies were silently zero on real music at listening volume (#1798 live finding, board #1857)** — `percussive_energy`/`harmonic_energy` reached the adaptive normalizer as raw masked *power* (magnitude², averaged over all 513 bins): ~1e-7 through a loopback capture at normal volume, below the normalizer's absolute span-collapse epsilon (1e-6), so `soft_norm` returned a hard 0 — while full-scale synthetic verification WAVs cleared the epsilon and made the pipeline look healthy. Every HPSS-gated visual (Cleave's shard bursts, the Tide splash gate, Vessel's HPSS behaviors) was therefore largely dormant in live use. An A/B on identical live drum-n-bass read old `percussive_energy` p50 = **0.000** vs fixed p50 = **0.635**. The energies are now **dB-mapped at the producer** (RMS → 20·log10 → −120..0 dB → 0..1, the bands' convention with a wider window for the /513-bin dilution — fulfilling the `features.rs` "dB-mapped" contract that was documented but never implemented): dB spans are volume-invariant, so the epsilon can never swallow them at any playback level. `harmonic_ratio` (Passthrough) is untouched — verified bit-identical distributions in the A/B. Cleave's gates were then retuned against **live** full-mix statistics (dense-groove percussive floor ~0.63/hit peaks ~0.85; full-mix `harmonic_ratio` lives in ~0.65..0.96, never the synthetic extremes): burst threshold above the sustained floor so needles articulate per hit instead of spraying continuously, pad-guard band moved from the middle of the musical range (where it cut strikes to a mean 0.35 *at drum hits*) to 0.90..0.98 raw, dominance crossfade remapped to actually travel with the arrangement, and needle lifetime/alpha/budget cut back from white-out to a radiant burst. Regression-tested (a −60 dBFS tone must still register) and live-verified via OSC A/B + screenshots.
- **Frame hitches no longer white-flash particle effects (#1796 live finding)** — the per-frame `dt` fed to every particle system and the fragment uniforms was raw wall-clock delta, so any stall (a real mouse click's input hitch, a window drag, an effect swap) integrated as one giant step: particles teleported past kill bounds and mass-respawned ("the flow starts over"), trail ribbons smeared across the whole screen for a frame (a full-screen white flash, amplified 16× by Tide's ribbons), and the emission accumulator dumped the entire stall's budget at once. `dt` is now clamped to 50 ms at the single computation site in `App::update` — a stall renders as momentary slow-motion instead of a flash, and normal frame times are untouched.
- **Obstacle field is no longer aspect-distorted — and horizontal surfaces actually bounce (#1790)** — four compounding defects, all shipped since the obstacle feature landed. **(1) Stretched UVs**: `obstacle_uv()` mapped clip space straight onto the full texture with no aspect term, so every obstacle map was stretched over the whole viewport — a 256×256 depth map on a 1920×1080 window was 78% too wide, skewing the very silhouette the upcoming Tide/Vessel effects are built around. **(2) Anisotropic normals**: `obstacle_normal()` stepped equal *UV* distances per axis, so on any non-square texture the central-difference gradient — and therefore every deflection angle — was skewed. **(3) Y-inverted normals** (found during this fix): the gradient was returned in texture space (y-down) but consumed by the collision response in clip space (y-up) — for a floor-shaped obstacle the "outward" normal pointed *into* the floor, so a falling particle failed the moving-into-surface test, was never reflected, and the surface-offset nudge pushed it deeper every frame: horizontal surfaces swallowed particles instead of bouncing them. **(4) Squashed depth**: the MiDaS path scaled a 16:9 webcam frame into the square 256×256 model input with independent axis factors, so the depth map was born distorted before the GPU stretched it again. Fixes: a new per-layer **Fit** mode (Fill = aspect-correct cover, **default**; Fit = letterbox; Stretch = legacy) drives an aspect-aware `obstacle_uv` — one new `obstacle_fit` uniform in the last spare pad slot, struct stays 832 B, all 16 sims covered via the shared lib; normals are now computed with equal screen-*pixel* steps per axis and returned in screen space (y-up), and the entire reflect/tangent response runs in screen space so bounces look right on any viewport (elasticity now preserves screen-space speed); the depth path letterboxes into the model input (edge-replicated padding), crops the output back to the valid region, normalizes over that region only, and uploads true-aspect maps (e.g. 256×144). The Fit choice round-trips through presets; pre-existing presets restore as Fill — the old stretch was a defect, not a choice, but Stretch remains selectable. Verified by a new (ignored, GPU-required) probe test that evaluates the real WGSL against a circle texture on a 1000×500 viewport: the circle must stay a circle in screen space, normals must be radially symmetric unit vectors — it fails under any of the four defects — plus letterbox/crop-normalize unit tests and preset serde coverage.
- **Scene transport bindings no longer re-fire every frame (#1791)** — a binding to `scene.transport.go`/`prev`/`stop` was level-triggered: `apply_binding_target` pushed a transport trigger on **every** frame the binding output sat above 0.5, so holding a MIDI pad — or wiring any source that stays high, like a sustained beat feature — advanced the scene timeline ~60 times per second instead of once per press; audio-driven cue control was unusable. The binding bus now keeps a per-binding rising-edge latch (`BindingRuntime::prev_above_threshold`, threshold 0.5 matching the old semantics — an upstream Gate transform composes as before) and `evaluate()` returns a `rising` flag alongside each output; only the `scene.transport.*` arm consumes the edge — every continuous target (params, layer opacity/enabled, postfx, particle, uniforms) stays level-driven and unchanged. The latch sits *after* the transform chain and deliberately freezes while a binding is disabled or its source drops out of the snapshot, so a briefly missing held-high source can suppress a trigger but never fire a spurious one. `evaluate()` also gained its first unit tests (via a new snapshot-level seam that doesn't need live MIDI/OSC systems): press/hold/release/re-press, post-transform edges, missing sources, disable/re-enable, and independent latches on a shared target.
- **`layer.{n}.blend` bindings can now reach all 10 blend modes (#1792)** — the blend arm fed the normalized 0..1 bus output straight into `BlendMode::from_u32(value.round() as u32)`, which only ever yields 0 or 1: a knob bound to blend mode toggled between Normal and Add and could never reach Screen, Color Dodge, Multiply, Overlay, Hard Light, Difference, Exclusion or Subtract (short of hand-adding a `Scale{9}` transform). New `BlendMode::from_normalized` spreads 0..1 evenly across `BlendMode::ALL` (0.0 → Normal, 1.0 → Subtract), clamping out-of-range input; the raw-integer OSC/WS paths already reached all 10 modes and are unchanged. Unit tests cover endpoints, every mode at its exact step value, rounding boundaries, and clamping.
- **Preset-scoped binding edits no longer silently lost (#1722)** — the debounced binding auto-save only ever flushed *global*-scoped bindings, so a preset-scoped binding you added or edited in the matrix reached the preset sidecar **only** via an explicit preset save; quit or switch presets first and the edit was gone, with nothing in the UI to warn you (the amber bar tracks a different, `PresetStore`-owned dirty flag the binding panel can't reach). Preset-scoped binding edits now mark the preset dirty, exactly like a param edit: a new `preset_scope_dirty` flag on the bus — set when a Preset-scoped binding is added / duplicated / edited / removed (scope is immutable after creation, so an id lookup is authoritative), polled each frame and forwarded to `PresetStore::mark_dirty`, cleared on explicit preset save/load — so the "unsaved changes" bar lights and the normal explicit-save sidecar flush persists them. Bindings now behave like the rest of the preset rather than auto-saving like globals. Separately, **quit now force-flushes pending global binding edits** (the confirmed-quit path previously saved only the pipeline cache, so a global edit made within the 1 s debounce window was lost on exit). New unit tests cover the per-scope flag and the flush.
- **Right-panel edits now round-trip through presets (#1721)** — hand-tuned **Lattice**, **particle-sim** (emit rate / burst / lifetime / speed / size / drag) and **Volumetric** parameters were applied to live runtime state but never saved into a preset, so they snapped back to the effect's `.pfx` defaults on the next effect reload or app restart — only the `.pfx` `inputs` params round-tripped. `LayerPreset` gains an optional `lattice` block (`LatticeParams`, including its embedded R3-marcher look) and a grouped `particle_sim` block, and `Preset` gains a top-level `volumetric` block — the global R3 mode is persisted like `postprocess`, not per-layer — with `LatticeParams`/`VolumetricParams` made `serde`-able. Values are captured on explicit preset save and re-applied **after** the effect reload resets the sim to `.pfx` defaults (the panel "Reset" still restores `.pfx` values — it reads a separate defaults snapshot that is never overwritten). Particle-sim and Volumetric edits now also light the amber "unsaved changes" bar, matching the Lattice/obstacle panels. Old presets without the new blocks load unchanged (`#[serde(default)]` → `None`), and a preset saved without a volumetric block disables volumetric on load so an earlier preset's mode can't bleed in. Round-trip + backward-compat covered by new serde unit tests.

### UI
- **Binding Matrix — live-workflow polish (UI overhaul 9/9)** — five quality-of-life changes for performing with the matrix. **(1) Templates land where you can see them**: applying a template while on the Global tab silently created its (Preset-scoped) bindings in the *other* tab — "nothing happened" from the user's view; applying now switches to the Effect tab (templates stay Preset-scoped by design: a Global `param.{effect}.*` binding would break silently on preset switch). **(2) MIDI Faders actually fires**: its `midi.*.cc.0.N` wildcard sources could never match (evaluation is exact-match on snapshot keys), so the template produced eight permanently-dead bindings; `apply_template` now substitutes the first live MIDI device's name for `*` (move a knob once before applying), and with no live device the wildcard survives and the new ⚠ no-signal chip flags each card — both paths unit-tested. **(3) Cards group by target** (untargeted last) instead of raw creation order, so multiple bindings driving one parameter sit together. **(4) Duplicate button** in the expanded editor clones a binding — source, target, transform chain — under a fresh id (unit-tested) and expands the copy: the fastest way to build "same source, three targets" stacks. **(5) Spotlight hover**: hovering a card brightens that binding's cables and dims all others, and the Effect/Global tabs show their enabled-binding counts. Deferred from the plan: double-click-a-source/target-to-create (the Learn button plus Duplicate cover the flow), and target-learn (no natural input event to learn from).
- **Binding Matrix — cables stay in their bay, popups stop closing the overlay (UI overhaul 8/9)** — two overlay-behavior bugs. **(1) Curve clipping**: the three columns scroll independently, but row positions were recorded for *every* row — including ones scrolled out of view — and the connection painter had no clip rect, so cables to off-screen rows drew straight across the header, footer and neighboring columns. Each column now records its visible band (`ui.clip_rect()` inside its scroll area), the cable painter is clipped to the union of the three bands, and an endpoint whose row has scrolled away is **clamped to the viewport edge as a faded stub** (mirroring how collapsed groups already snap their cables to the group header) with its flow dot suppressed; a cable whose both ends are off the same edge isn't drawn at all. **(2) Popup-aware dismissal**: any click outside the panel rect closed the overlay — including the click that *selects an item* from the 350 px source picker, the grouped target combo, or the "+ Add" transform popup when they overhang the panel edge — and Escape closed the whole overlay instead of the open popup. Both paths are now guarded by `egui::Popup::is_any_open`, sampled before the frame's widgets so the dismissing click itself is covered; Esc with a popup open now closes just the popup (egui's own behavior), Esc again closes the matrix.
- **Binding Matrix — cards get names, a leak dies, ids and saves behave (UI overhaul 7/9)** — four fixes to the matrix overlay. **(1) Collapsed cards now say what they are**: the compact row rendered only the enable dot, transform pills and I/O meters — an unnamed binding read as just "~ Smooth" — so it now leads with the binding's name (or the auto "Source → Target" the expanded editor already used as hint text), char-safe-truncated with the full text on hover, plus a **⚠ no-signal chip** when an enabled binding's source key isn't in the live snapshot (unplugged controller, un-substituted template — previously such bindings just silently did nothing). **(2) Per-frame memory leak fixed**: multi-layer target group labels were `Box::leak`ed on every rebuild — and `build_target_options` ran **twice per frame** while the overlay was open — an unbounded leak; `TargetOption.group` is now a `Cow<'static, str>` and the options build once per frame, shared by both columns. **(3) Idle-expanded cards no longer rewrite the bindings file**: the expanded editor wrote source/target/name/enabled back through `get_binding_mut` (which marks the bus dirty) every frame, so the debounced save rewrote `global-bindings.json` every second for as long as a card sat open; the writeback now fires only on an actual change. **(4) Duplicate-id repair**: the `b_{n}` id counter is per-file, so a preset sidecar authored in another session could reuse an id a global binding holds — the runtimes map silently collided and deleting either card removed **both**; preset loads now go through `merge_preset_bindings`, which re-ids collisions (sidecar self-heals on next preset save), unit-tested. Bonus fix found while testing: bindings **unit tests wrote the real `~/.config/phosphor/global-bindings.json`** (a test bus is empty, `remove_binding` saves immediately — a plain `cargo test` clobbered your global bindings with an empty list); persistence saves are now no-ops under `cfg(test)`.

A voxel grid of cells (dead / alive / dying) evolves by 3D totalistic **birth/survival rules** counting neighbours in a 3×3×3 Moore (26) or Von Neumann (6) neighbourhood, producing self-organising volumetric structures — growing crystals, pulsing organisms, expanding shells, collapsing caverns. Unlike the 2D reaction-diffusion / physarum fields, **the CA state *is* the 3D volume**: the only inherently-volumetric simulation in the library. Ships as **eight selectable built-in effects** (Lattice Clouds / Crystal / **Pyroclastic** / Architecture / Coral / Builder / 445 / Pulse), each a `.pfx` that bakes one rule — hosted exactly like Turing and Polycephalum via an optional `particles.lattice` def block that builds the sim in `ParticleSystem::new`, so **you pick it from the effect browser** (no global toggle, and no dependency on having a particle effect selected — the earlier build wrongly inherited R3's toggle shape even though Lattice reads no particles). `gpu/lattice.rs` `LatticeSim` owns ping-pong `array<u32>` state buffers plus its own resizable `r32float` 3D density texture, and **reuses the R3 ray marcher unchanged** via a shared `create_raymarch` builder + `VolumetricUniforms`/`VolumetricParams` (the tuned orbiting camera, Beer-Lambert march, self-shadow, soft envelope and Reinhard tone-map are one source of truth, not a fork — the only R3 edits are additive helper exposure). - **Volumetric Mode — particle density 3D ray marching (Viz R3, tight-core V1)** — a global rendering toggle that renders any particle effect as a continuous fog/nebula instead of discrete dots: the app's **first true 3D-volumetric look**, and the top pending item on the next-gen roadmap (it is the hard prerequisite for the flagship Lattice 3D-cellular-automata effect that follows). Three GPU steps per frame, run **inside `ParticleSystem`** — the only place the private SoA particle buffers are reachable, exactly why `ComputeRasterizer` lives there too, so a free-floating post-compositor chain (which only ever sees a flattened 2D HDR target) was never an option. (1) **Scatter** (`volumetric_scatter.wgsl`, one thread per alive particle): particles are fundamentally 2D (`pos_life.z` is a spin angle, not depth), so a **stable Z is synthesized from a PCG hash of the particle index** — hashing the *index*, not the thread or time, so a particle's depth is temporally coherent and the flat cloud gains a fixed 3D structure the moving xy sweeps through — then `(x, y, z)` is projected into the unit cube `[-1,1]³` and **`atomicAdd`s fixed-point density into a `u32` voxel buffer** (`textureStore` has no atomic form, and float storage textures can't be atomic targets without an unrequested feature, so additive accumulation must go through a buffer — the `ComputeRasterizer` shape). (2) **Resolve** (`volumetric_resolve.wgsl`, `@workgroup_size(4,4,4)`): maps the raw voxel counts through a **saturating normalise** (`1 − e^(−gain·occupancy)`, so density is bounded and independent of particle count — no white-out on dense effects), applies a **3×3×3 box blur** (what actually turns discrete deposits into smooth fog), and `textureStore`s into the samplable 3D density texture. (3) **Ray march** (`volumetric_raymarch.wgsl`, fullscreen fragment): builds an **orbiting camera ray inline** (no view/projection matrix — the strata.wgsl technique, so the shared 400-byte shader ABI is untouched; volumetric params ride their own bind group), bounds the march to the cube with a **slab ray/box test**, then accumulates Beer-Lambert transmittance with FBM detail breakup and a cosine emission palette — **shaded by a secondary self-shadow light march** (nearest-voxel taps toward a fixed key light, so dense regions cast shadows and the cloud shows real 3D form instead of the flat uniform glow self-emission alone produces at any emission level), inside a **soft envelope** that fades density toward the cube faces (so a filled field reads as a cloud, not a lit box), and **Reinhard-tone-mapped** so emission lifts brightness without clipping to white — compositing premultiplied over the scene. The density texture **must be `r32float`**, verified against the wgpu-27 format tables and the device's `Features::empty()` config: `r16float` has *no* storage caps at all without `TEXTURE_ADAPTER_SPECIFIC_FORMAT_FEATURES`, and `r32float` — which also happens to match Lattice's downstream `texture_storage_3d<r32float>` contract — is **not hardware-filterable** without `FLOAT32_FILTERABLE` (also unrequested), so the marcher samples via **manual trilinear `textureLoad`** (8 fetches + `mix`, no sampler); binding a `Linear` sampler would have aborted bind-group creation at runtime. The density texture + marcher are deliberately factored as the **reusable core** (`density_storage_view()` is exposed for a future producer): particle scatter is just one density source, and Lattice will later write that texture directly and reuse the marcher unchanged. Integration replaces the normal particle render only while enabled — a branch in `ParticleSystem::render` before the compute-raster branch, and the scatter/resolve replacing the compute-raster dispatch — so billboard and compute-raster effects alike can toggle it. The renderer is **lazily built on first enable** (no cost when volumetric mode is never used) via the per-frame layer-update loop, which already has `&mut ParticleSystem` + device access; the global toggle applies to the **active particle layer** (multi-layer is deferred). Control surface: a **Volumetric (R3)** side-panel section (enable + density / absorption / volume-depth / emission / detail / palette / camera / march-steps sliders, plus a **Reset to defaults** button) and **OSC** — `/phosphor/volumetric/enabled` and `/phosphor/volumetric/{param}` (mirroring the post-process toggle path). Audio-reactive: `beat_phase` pulses density, `kick` bursts emission, `rms` drives absorption, `time` animates the detail FBM — read from the audio the particle system already carries, so **no new audio plumbing**. Runtime-smoke-verified headless on **lavapipe** (software Vulkan, nested Xephyr — RTX and system audio untouched): on OSC enable the renderer initialised (`r32float 64³` volume), meaning all three WGSL shaders compiled on-GPU and the 3D `STORAGE` texture + every bind group (the `WriteOnly` storage-3D entry and the `Float{filterable:false}` sampled-3D entry with no sampler) validated and every pipeline built; the app rendered through enable → live param tuning → disable with **zero wgpu validation errors, zero panics, no crash**, clearing the two headline risks (3D storage on lavapipe; the r32float non-filterable trap). - **Polycephalum — 12-species slime-mold warfare keyed to the chroma vector (Viz V1)** — the first effect that *consumes* the full 12-element chroma vector (board finding #1490: chroma/flatness/rolloff/bandwidth/zcr were used by ZERO effects — a deep DSP pipeline nothing rendered). - **Spectral contrast + delta-MFCC timbre dynamics (A16)** — fills the final eight reserved shader-ABI-v3 slots (`contrast_0…contrast_5` / `contrast_mean` / `timbre_flux`) with **zero further ABI change** (DSP + policy only), completing the v3 tail the batched bump (#1634) opened after A14 and A15. Two gaps closed: MFCCs captured a frame's *static* envelope but nothing measured its *motion*, and `flatness` measured tonality only *globally* — nothing scored the per-band peak-vs-valley "grit" axis that separates a clean sine from a saw in the same octave. **Spectral contrast** (Jiang 2002 / librosa) is a new `FftAnalyzer::spectral_contrast()` over six octave bands (200-400 … 6400-Nyquist Hz): per band, `dB(mean top 2%) − dB(mean bottom 2%)` mapped 0-60 dB → 0..1, plus their mean. It reads the **large (4096-pt, 10.8 Hz/bin)** magnitude for *every* band — not the medium spectrum A14 uses — so even the 200-400 Hz octave has ~18 bins to take a 2% quantile over (the medium gives ~4); the top/bottom counts are `ceil(0.02·n).max(1)` (librosa's `alpha`) and a lone peak over a silent floor saturates to 1. The seven values are producer-mapped 0..1 → **Passthrough**, silence-gated **inside** the analyzer (the normalizer won't gate a Passthrough field, and the noise floor has its own spurious peak/valley structure). **Delta-MFCC** is a new `audio/timbre.rs`: `DeltaMfccAnalyzer` keeps a causal 5-frame ring of the (pre-normalization) MFCC vector and fits a per-coefficient least-squares slope — `Σ(t−t̄)·x_t / Σ(t−t̄)²`, weights `[−2,−1,0,1,2]/10` for the 5-frame window — adding **zero latency** (the newest frame is one window edge). The 13 slopes feed **bindings-only `audio.dmfcc.0..12`** sources (the `audio.mel.N` transport pattern: a new `dmfcc` field on `AudioFrame` → `AudioSystem::latest_dmfcc` → `collect_dmfcc_bands` in the bus), staying **out of the uniform to save 52 bytes**. **`timbre_flux`** is the L2 norm of the slope over **coefficients 1..12 only** — excluding MFCC[0] (the log-energy term) keeps it a pure timbre-*shape* signal orthogonal to `flux`: a constant-loudness filter sweep fires it, a plain volume change (which scales all mel energies equally → moves only MFCC[0]) does not. It is a raw excitation of unknown scale, so — unlike the reserved Passthrough placeholder — it is **Adaptive**, percentile-ranged and silence-gated like its sibling `flux`, with `flux`'s fast `ar(0.005, 0.06)` smoothing; the ring clears on the A10 silence flag so a rest can't spike the slope on the next note. Broadcast over OSC TX (`/phosphor/audio/{contrast_0..5,contrast_mean,timbre_flux}`; `dmfcc` is bindings-only, not echoed). - **Monophonic pitch tracking — f0 + confidence (A15)** — fills the two reserved shader-ABI-v3 slots (`pitch` / `pitch_confidence`) with **zero further ABI change** (DSP + policy only), the second fill on the v3 tail the batched bump (#1634) opened. The engine could name the *key* (A11) and the *pitch classes* present (chroma) but never the actual **melodic f0** — so no effect could follow a lead line or a bassline's note. A new `audio/pitch.rs` runs **classic YIN** (de Cheveigné & Kawahara 2002) directly on the analyzer's raw, un-windowed 4096-sample time-domain window (reused via a new `FftAnalyzer::time_domain()` accessor — no extra FFT, and DC-immune, since YIN's difference function cancels any constant offset): the **difference function** `d(τ) = Σ (x[j]−x[j+τ])²` over a 2048-sample integration window, its **cumulative mean normalized** form `d'(τ)` (which suppresses the octave-too-high error), an **absolute-threshold** pick (the first dip below 0.15 followed to its local minimum — taking the *first* qualifying lag, not the global minimum, biases toward the fundamental over 2T/3T), and **parabolic interpolation** for a sub-sample period (≥2 periods fit the window down to ~43 Hz, so basslines resolve). f0 is mapped to a **0..1 log-frequency** — 55 Hz (A1) → 0.0, 1760 Hz (A6) → 1.0, a clean 5 octaves so an octave is always 0.2 — and **held through unvoiced gaps** (confidence gated to 0) so a pitch-keyed visual doesn't snap to the lowest note on every rest. `confidence` is the YIN periodicity `1 − aperiodicity`. - **Harmonic/percussive split — percussive / harmonic energy + ratio (A14)** — fills the first three shader-ABI-v3 slots (`percussive_energy` / `harmonic_energy` / `harmonic_ratio`) with **zero further ABI change** (DSP + policy only), the first fill on the v3 tail the batched bump (#1634) opened. ## v1.9.0 — 2026-07-17

### Added
- **Follow the default output device when it changes (A9b)** — A9 below reconnects a capture device that *dies*, but the most common real-world case on Linux is not a death: **switching your system output mid-set**. PipeWire silently **migrates** our `pa_simple` stream to the new default sink's monitor — reads keep flowing at 43/s, `callback_count` keeps advancing, no error is ever raised — so A9's watchdog is *structurally* blind to it and the app goes on capturing the **wrong device** while looking perfectly healthy (finding #1614, runtime-verified: destroying a null sink under the running app moved its stream to a different live source and `poll_health` never noticed). The sink name is the only signal there is, so it is now polled: a background worker asks `pactl get-default-sink` every 5 s and compares `<sink>.monitor` against the live device name; on a change the capture reopens onto the new default. **This needed no new open path** — `current_target()` already returns `None` on the native backend, so the existing `adopt(Err) → worker → adopt(Ok)` cycle re-runs `find_monitor_source` and lands on whatever is default *now*; A9b is a trigger and nothing more. Two placement details are load-bearing. The board task proposed hanging the poll on "the existing 5 s background devscan thread", but no such thread exists: `list_devices` is **pull-driven** and its only caller is the audio panel, so a detector built there would have worked **only while that panel was open** — useless for a fullscreen set. It hooks `poll_health` instead (driven every frame), and must run **before** that function's healthy-path early return, since a migrated stream is precisely the case where callbacks keep advancing — placing it after would have made it dead code in the one situation it exists for. `pactl` forks a subprocess, so the poll runs on a worker and the render thread only ever reads the cached answer. Gated on `using_native_backend` for two independent reasons: the device name holds a PA monitor source **only** on that path (it is a cpal device name otherwise, so the comparison would be nonsense), and a user who explicitly picked a device did not ask to follow the system default. It **reuses the `auto_reconnect` setting** rather than adding a knob — choosing the native backend already means "capture whatever my system is playing", and following the default sink is that choice's literal meaning. A sink change enters the episode at attempt 1, so a new sink that refuses to open still gets A9's 1/2/4/8 s backoff ladder rather than giving up on first failure, and `Phase::Opening` stops the reopen re-firing on every rendered frame. Linux/Pulse only; the WASAPI equivalent is a follow-up (finding #1616). Verified end-to-end against a real running app: opened on a null sink, then `pactl set-default-sink` away → followed within ~4 s, old backend reaped in 15 ms, **fully reconnected 34 ms later**, with `pactl list source-outputs` confirming the stream had actually moved; with auto-reconnect **off**, zero follow attempts; and A9's own death path (`pw-cli destroy`) still reconnects unchanged. (board #1617)
- **Audio watchdog auto-reconnect with off-thread teardown (A9)** — the stall watchdog was **detection only**: it raised a toast and left you to re-pick the device by hand (see the v1.8.0 "Visuals settle on audio stall" note, which this supersedes). The blocker it documented is real — tearing down a stalled backend drops `CaptureBackend`, whose `Drop` **joins** a capture thread that may be blocked in a timeout-less `pa_simple_read`, freezing the render thread for as long as the stall lasts. The fix is to stop joining on the render thread: a confirmed death now hands the backend to a **detached reaper** (`audio/reconnect.rs`) that runs the `Drop` — and its join — on its own thread. It is a *deferred* reaper, not a leak: every capture loop re-checks its shutdown flag at the top of each iteration, so the join completes the moment the source produces anything; the pathological case costs one parked thread and the show keeps running. Only `_capture` is reaped — the **analysis thread is still joined inline**, because it only ever polls its flag and sleeps 10 ms and so is bounded even mid-stall. That is load-bearing rather than merely simpler: `RingBuffer::push` writes through a raw pointer under a single-producer contract, so sharing `recording_ring` across a reopen (below) is only sound because the old thread is provably gone before the new one starts. **The trigger is a positive death signal, not a frozen callback count.** The board task framed the ambiguity as WASAPI-specific ("loopback delivers no packets while nothing plays"), but a PulseAudio **monitor** is the same shape — a suspended sink stops feeding it — so reconnecting on a 10 s freeze would fire on every idle pause on *both* native backends. Each backend now publishes `capture_failed` when its thread exits on an error (cpal's `err_callback` on `DeviceNotAvailable`/`StreamInvalidated` — deliberately **not** `BufferUnderrun`, a glitch on a live device; WASAPI when the COM loop returns `Err`, guarded on `!shutdown` so a failing `Stop()` during teardown can't fake a death; Pulse after 10 consecutive read errors), paired with a `silence_delivers_data` policy bit that is **true only for cpal**, whose callbacks run on the device clock and deliver zeros through silence. Both live on `OpenedBackend` and are set in `open_backend`'s already-`#[cfg]`-gated arms, so nothing downstream knows what platform it is on. The reopen is **asynchronous**: `open_backend` forks a `pactl` subprocess and makes two blocking PulseAudio round-trips, and a dead server — the likeliest cause of the stall being answered — bounds neither, so running it inline would put an unbounded wait back on the render thread and undo the reaper. The whole cycle is `adopt(Err) → worker → adopt(Ok)`, reusing the existing "no device" state as the in-between (its stale-feature decay already carries the visuals across the gap) — no new lifecycle state, and no generation counter needed, since `pending_open` solely owns the receiver: dropping it orphans the worker, whose `OpenedBackend` then drops on its own thread, a free second reaper. A pure `ReconnectState` (16 unit tests, no device, no clock of its own) runs attempt 1 immediately then backs off 1/2/4/8 s to 5 attempts; an `Opening` phase stops the reopen re-firing every rendered frame, and a 5 s **probation** window defines what a *failed* attempt is — necessary because `AudioCapture::new_with_device` **silently falls back to the default device** when the named one is missing, so `Ok` alone proves nothing. A death inside probation consumes an attempt rather than starting a fresh episode, which is what stops a backend that dies on open becoming a hot reopen loop. Gated by an **`auto_reconnect`** setting (Settings → Auto-reconnect audio), which needs `#[serde(default = "default_true")]` rather than the bare `#[serde(default)]` every other field in `SettingsConfig` uses — `bool`'s `Default` is `false`, which would have silently shipped the feature **off** for every existing `settings.json`, with green tests. A new **AUD** status-bar dot reports live/quiet/reconnecting (with an `n/5` counter)/failed. Verified end-to-end against a real running app on PipeWire by destroying its capture node mid-session: read errors at T+0, death declared after 10 consecutive errors (~0.9 s), `phosphor-audio-reaper: released in 0ms`, reopened onto the fallback default, first data at **T+960 ms** — one episode, attempt 1, no loop. **A finding that reframes the Linux story** (filed as follow-up): destroying a *source* does **not** kill a `pa_simple` stream under PipeWire — it **migrates** it to the new default, so reads keep flowing at 43/s with no error, no stall and no death signal, and the app silently captures the wrong device. That whole class of failure is invisible to any watchdog and belongs to the split-out default-sink-change work rather than here — **A9b, above, is that work, and ships alongside this**; it also makes the Pulse death path fire mainly on genuine server/stream death. (board #1460)
- **Locally-advanced bar phase (A8b)** — A8 left `bar_phase` on `Hold`, so while `beat_phase` ran smooth at render rate the *bar* sawtooth kept its 86.1 Hz **stair-step**: on a 60–144 Hz display it repeated ~10% of frames outright and its step sizes varied 6× frame to frame, which any bar-synced motion inherits. It now runs through the same first-order PLL, advanced every render frame and wrap-aware soft-resynced to the tracker. A8 deferred this because locally advancing the bar looked like it needed the downbeat tracker's `meter`, "whose mid-bar hysteresis flip changes phase rate *and* wrap point at once" — but that blocker was an artifact of choosing `meter`. `DownbeatTracker` rewrites `bar_duration` **only inside the branch that also zeroes `bar_phase`**, so the frame carrying a new bar length carries `bar_phase == 0` *in the same message*: publish **`bar_duration` instead of `meter`** and the rate change and the wrap become one atomic event the render thread can never observe mid-bar. It is also the only safe choice — `meter` changes on any beat (the low-confidence path re-assigns bar phase every beat with **no** hysteresis), so a render-derived rate would be *more current* than the phase it chases, and on a lost tempo lock (`bpm → 0`, `meter` still 4) it would **freeze** the local phase while the tracker's kept ramping. The tracker now publishes the sawtooth and its denominator from **one gate** (`bar_clock`), so "`bar_duration > 0` ⟺ the bar clock is live" holds by construction across the thread boundary rather than by each side re-deriving it; the render holds the pair from the **older** bracketing frame, so one playhead crossing swaps the phase reset and the rate together. Fixing this exposed a **latent PLL bug in A8**: solving the loop shows a soft correction outruns the advance — running the sawtooth *backwards* — whenever `|err| > τ·rate`, and A8's flat quarter-cycle `PHASE_SNAP` only ever cleared that bound by accident of tempo (τ·rate = 0.30 at 120 BPM vs a 0.25 threshold). At a bar's 0.5 Hz the bound is **0.075**, so the tracker re-assigning which beat is the "one" would sweep `bar_phase` backwards for ~180 ms and land the visual "one" a quarter-bar late — worse than the stair-step. The threshold is now **derived from the rate** (`(PHASE_RESYNC_TAU · rate).min(PHASE_SNAP)`), making "the phase only ever rises or wraps" a property of the loop rather than of the tempo; this is **byte-identical for `beat_phase` at ≥100 BPM** and, as a deliberate side effect, closes the same reversal band A8 left open for `beat_phase` **below** 100 BPM. **No shader-ABI change** (`NUM_FEATURES` stays 61; `bar_duration` rides `AudioFrame`, not the `#[repr(C)]` `AudioFeatures`) and no consumer changes — OSC, bindings and shaders read the smoothed value for free. Verified A/B end-to-end against a synthetic 32-bar 120 BPM 4/4 signal (accented downbeats + per-bar chord changes) through a PulseAudio null sink, same audio both runs: `bar_phase` duplicate frames **10.38% → 0%**, step-size coefficient of variation **0.493 → 0.085 (5.8× more uniform)**, mean step landing on the render-rate prediction (0.00838 vs 0.00835 expected), and the tracker's own re-alignments now taken as **2 isolated snaps instead of 9** — while bar-period and lock behaviour are unchanged, confirming the render side reproduces the audio thread rather than editorialising it. A8b also made one long-standing asymmetry impossible to ignore: `DownbeatTracker` had no silence gate, so unlike `beat_phase` (pinned at 0) `bar_phase` free-ran through silence — A8b reproduced that faithfully rather than inventing a freeze the audio side lacked, but doing so *smoothly* is what finally made it worth deciding on. It was decided: see the `bar_phase` silence fix under **Fixed** below, which ships with this. (board #1554)
- **Render-rate feature interpolation + locally-advanced beat phase (A8)** — the render thread took the newest 86.1 Hz analysis frame as-is, so on a 120–144 Hz display ~28% of frames reused an identical feature vector; worse, `beat_phase` is a sawtooth that only moved when a frame landed, so phase-locked motion visibly **stair-stepped**, and a dropped frame (the channel is `bounded(4)`, drop-on-full) read as a phase **pop**. New `audio/interp.rs` keeps a short ring of **sample-clock-timestamped** frames (the A5 clock, #1456, now carried on `AudioFrame` — *not* on the `#[repr(C)]` `AudioFeatures`, so **no shader-ABI change**) and a render playhead held **1.5 hops (~17 ms) behind** the audio clock: it free-runs by render `dt` and is gently slewed toward the target (capped at ±10% of real time, so it can never stall or reverse), then the two frames bracketing it are blended per a new **`InterpPolicy` schema column** — 52 continuous slots **Lerp**, 9 **Hold**. The `Hold` set is deliberately *not* derivable from the existing `smooth.bypass`, and the two disagree in both directions: `dominant_chroma` is smoothed yet must Hold (lerping an argmax index reads out a pitch class that was never detected), while `bpm` Holds on decay yet lerps fine. `beat_phase` is then replaced outright by a **first-order PLL** — advanced every render frame at `dt·bpm/60` and wrap-aware soft-resynced to the detector (the rates agree by construction: `period == 60/bpm` exactly), snapping only on genuine desync (>¼ beat) and following the detector's freeze under silence. The beat/downbeat/drop **atomic counter-latch is untouched and still runs last**, so pulse timing is unaffected — the added ~17 ms is presentation latency only, on a chain already carrying ~93 ms of FFT window. Interpolation is skipped (falling back to the held frame, i.e. the old behaviour) at startup, on a device switch — where the sample clock restarts at 0 and the timestamp jumps *backwards*, so the ring is dropped and the playhead re-seeded — and on the >250 ms stall path, which is **load-bearing**: the stall decay mutates the held frame, so leaving the ring live would keep serving the last *undecayed* frame and silently freeze visuals loud on a dead device. `spectrum`/`mel` are deliberately not interpolated (a documented ~17 ms skew; 512-bin blending per render frame is real CPU for no visible gain). Verified end-to-end against a 120 BPM signal through a PulseAudio null sink: `beat_phase` now advances on **every** render frame (0 duplicate samples, was 5.3%), delta uniformity improved **10×** (cv 1.39 → 0.13, max jump 0.77 → 0.07), tempo locks at **1.998 wraps/sec vs 2.000 expected** over 40 s with no drift, beats still fire exactly 2.000/sec, the phase freezes at 0 under silence, and yanking the capture device decays `rms` 0.35 → 0.0 rather than freezing loud. `bar_phase` has the same stair-step and gets `Hold` here (which already stops its wrap being lerped into a backwards sweep through 0.5); locally advancing it was deferred to A8b, which shipped below. ### Changed
- **Spectral feature correctness pass — centroid / flux / flatness / rolloff (A4)** — the four spectral-shape features were each subtly wrong. **Centroid** was magnitude-weighted on a linear-Hz axis and included the DC bin, so it lived in the top octave and was unstable near silence; it is now a **power-weighted mean of log2(frequency)** (skipping DC) mapped to 0..1 over a musical 40 Hz–18 kHz range — a usable brightness fader, held steady below the silence gate by its FixedRange policy (A2). **Flux** was an unnormalized linear-magnitude sum over the whole spectrum, so it doubled when the volume doubled (a second RMS); it is now a **level-invariant** per-bin log-magnitude half-wave difference averaged over bins (with a −60 dB floor so noise-floor bins, where per-bin log is wildly unstable, don't dominate), so it measures *change* not level and the Adaptive percentile policy ranges it. **Flatness** skipped tiny bins from both means (biased up in sparse spectra) and was dominated by the dense HF region; it is now **Wiener entropy over the 26 mel bands** (reusing the MFCC filterbank, every band counted), cleanly separating tonal pads (low) from noise sweeps (high). **Rolloff** now skips DC and its 85% is a named const (`ROLLOFF_PERCENTILE`) instead of a magic number. The six spectral-shape features (`centroid` / `flux` / `flatness` / `rolloff` / `bandwidth` / `zcr`) are now also **broadcast over OSC TX** (`/phosphor/audio/*`) — previously only `kick` among them was emitted, so they were unreachable to external tooling and unverifiable end-to-end. No shader-ABI change. (board #1455)
- **Kick detection — single detector-owned normalizer (A3)** — `kick` was **double-AGC'd**: the analyzer normalized a *linear*-magnitude 30–120 Hz flux by a peak-hold `kick_max` (0.999/frame ≈ 10 s hold), and then — because slot 8 was `Adaptive` — the feature normalizer stretched it a *second* time, so the kick was nearly always saturated on quiet material and its scale depended on two interacting auto-levelers. It is now computed **once**: a **log-magnitude** half-wave flux (the same level-invariant form as the SuperFlux onset detector, so a bass *change* registers regardless of absolute level) normalized by its **own long-term P95** (`FftAnalyzer::kick_envelope`, reusing the A2 `PercentileWindow`), and marked **Passthrough** in the schema so the feature normalizer no longer touches it. The envelope is **gated on the A10 perceptual silence flag** (momentary < −55 LUFS) and skips its window push during silence, so log-flux of the noise floor can't manufacture kicks and the P95 is frozen for the next active passage — a kick after a quiet break isn't over-scaled. The 30–120 Hz magnitudes are also **floored (−60 dB) before the log**, so the near-silent low bins under a bassless lead or a high sweep read a constant zero instead of jittering into false kicks (verified end-to-end: a pure 1 kHz/440 Hz tone and a 200→8 kHz sweep now read kick ≈ 0, while a real kick and broadband noise still fire). Result: kick-bound strobes stop firing on hi-hat bleed, bassless leads, and quiet passages, and no longer saturate on loud sustained bass. Deleted the dead `kick_max` / `prev_kick_flux` analyzer state; no shader-ABI change; the optional binary `kick_pulse` is deferred to the next batched ABI bump (decision #1504). (board #1454)
- **Gated percentile normalization + per-feature policies (A2)** — the adaptive normalizer was a single symmetric running-min/max with one global decay applied to *every* feature, which (a) stretched quiet-room noise to full scale so visuals danced to silence, (b) let one transient spike squash everything for ~2 s while the max decayed (pumping), and (c) gave signed MFCCs and already-max-normed chroma the same min/max treatment as energy bands. Replaced `AdaptiveNormalizer` with a **`FeatureNormalizer`** that dispatches on a per-slot `NormPolicy` (extended with `FixedRange` and `ZScore`): **Adaptive** (bands / rms / flux) is now **gated percentile ranging** — `(v − P5) / (P95 − P5)` over a ~4 s windowed history with a soft knee above P95, so one outlier no longer defines the top of the range and the window length *is* the spike-recovery time (the old "decay after spike" heuristic is gone); on the **A10 perceptual silence gate** (momentary < −55 LUFS) energy features read 0 with the window **frozen**, so a quiet room can't be auto-ranged up. **FixedRange** (centroid / flatness / rolloff / bandwidth / zcr — features A4 puts in a known 0..1 range) clamps and **holds its last value** through silence instead of dancing. **ZScore** (the 13 signed MFCCs) standardizes each coefficient by a running mean/variance and maps ±3σ → 0..1 through tanh, so excursions read symmetrically about 0.5. **Chroma + dominant_chroma** move to Passthrough (the CQT already max-norms them). New `audio/ranging.rs` holds the shared windowed-percentile primitive (also reused by the A3 kick). Clean replacement — no A/B setting, no shader-ABI change. (board #1453)
- **Unified band scaling — all seven bands in one dB domain (A1)** — the 7 frequency bands were computed in **two incompatible families**: `sub_bass`/`bass`/`low_mid`/`mid` as linear RMS magnitude and `upper_mid`/`presence`/`brilliance` as dB(−80..0), so a visual bound to `band.0` and one bound to `band.5` behaved like different species and the adaptive normalizer saw wildly different input dynamics. A new **`band_scale`** setting (`Unified dB` default, `Legacy` opt-out) puts all seven bands in one **dB(−60..0)** domain with a **+3 dB/oct equal-loudness tilt above 2 kHz**, so the bands are directly comparable (each keeps the FFT resolution best suited to its range). This sharpens everything downstream that reads the bands — the normalizer, A18's timbre vector and sub-bass-withdrawal logic, A6's partitions. **`Legacy` reproduces the exact pre-A1 behavior** for presets tuned to the old feel; the setting lives in `settings.json` and a **Settings → Band Scale** dropdown that applies live (rebuilds the capture pipeline). Deferred to a follow-up (#1452 part b): exposing 32 mel bands as `audio.melband.0..31` binding sources. No shader-ABI change. (board #1452)
- **SuperFlux onset detection — closes the 250–500 Hz snare gap (A6)** — the onset stage was a weighted spectral flux over four fixed bands (20–80, 80–250, 500–2000, 2000–4000 Hz) with a **hole at 250–500 Hz** (snare bodies, toms, male vocals) that also false-triggered on vibrato and pitch slides. It is now **SuperFlux** (Böck & Widmer, DAFx 2013): a 64-band contiguous log-magnitude filterbank flux (built on the 4096-pt spectrum) whose reference frame is passed through a **frequency maximum filter** (±1 band), so a partial can drift slightly between frames without registering an onset — killing the phantom onsets on wobble synths and vibrato — while the contiguous bands cover the old snare gap. The bands are grouped into low/mid/high partitions whose mean flux is weighted (0.60/0.28/0.12) to preserve the previous kick/snare/hat balance, and the existing median + 2·MAD adaptive threshold + ceiling logic and onset-strength normalization are unchanged, so beat/tempo tracking downstream is undisturbed (all BPM-convergence tests still pass). The detector now shares the **A10 perceptual silence gate** (momentary < −55 LUFS) instead of a raw-RMS threshold, so silence is judged the same way across devices and content. Relies on the fixed hop (A5) for a well-defined frame-to-frame max filter. (board #1457)
- **Deterministic fixed 512-sample analysis hop (A5)** — the audio thread previously slept 10 ms and analyzed whatever samples had arrived (an effective hop of ~441 samples ± scheduler jitter that grew under load), so spectral-flux/onset amplitudes varied with hop size and the tempo estimator had to guess its own frame rate from wall-clock timestamps. It now accumulates capture reads into a FIFO and runs one analyze→normalize→beat→downbeat→smooth frame per **exactly `ANALYSIS_HOP` (512) samples**, looping when a read yields several hops (e.g. catching up after a stall). Frame rate is now exactly `sr / 512` (≈86.1 Hz @ 44.1 kHz), passed to `BeatDetector::new`, so the tempo estimator's **runtime frame-rate re-estimation is deleted** (it existed only to compensate for the jittery hop). Frames are timestamped from a **sample clock** (`samples_consumed / sr`) instead of `Instant::now()`, so timing is exact even when a burst of hops is processed in one wakeup. The 4096-sample analysis window's 87.5% overlap is unchanged; all beat/tempo tests pass. Prerequisite for SuperFlux onset (A6, whose time max-filter needs a fixed hop) and clean ~10 Hz decimation in structure detection (A18); also feeds sample-clock timestamps to A8. (board #1456)
- **Audio feature schema (single source of truth)** — added `audio/schema.rs` with an ordered `FEATURES` table that owns each feature's normalization, smoothing, and stale-decay policy. The adaptive normalizer (passthrough set), the attack/release smoother (previously a hand-ordered 46-entry table), and the stale-feature decay (previously a `BPM_INDEX = 18` literal + a special-cased `beat = 0`) now all read their per-feature policy from that table instead of hardcoding positional indices, so adding a feature no longer risks silently shifting indices out from under those stages. A `const` length assertion plus a layout-guard test pin the table to the `#[repr(C)]` struct order. Behavior is unchanged (all existing normalizer/smoother/decay tests pass). Prerequisite for the batched audio-feature additions ahead. (board #1451)

### Fixed
- **`bar_phase` swept through silence while `beat_phase` sat frozen (#1598)** — the two phase clocks disagreed about what silence means. `BeatDetector` takes the A10 perceptual silence gate and pins `beat_phase` at 0; `DownbeatTracker` never took it at all — so through a quiet passage beats stop firing, `last_downbeat_time` and `bar_duration` freeze, and the bar clock, being a pure function of the audio timestamp, **keeps ramping on regardless**. The result was an incoherent pairing: verified live on a silent monitor, across 361 samples `beat_phase` held a single value (pinned 0) while `bar_phase` swept the full 0→1. Pre-existing since A12 (#1463) and invisible while the bar sawtooth still stair-stepped, but A8b (above) makes that sweep *smooth* — and a smoothly sweeping bar against a dead beat is exactly the kind of thing a bar-synced visual inherits. The tracker now takes `loud_silent` (already in scope at the call site, one line away from the beat detector that had it all along) and pins `bar_phase` to 0, with the render-side interpolator gaining the matching `frozen` gate its beat-phase counterpart already had. **The bar *rate* keeps being published while pinned**, deliberately: dropping `bar_duration` to its 0 sentinel would have needed no render-side change at all — the "no bar clock" branch would have swallowed it — but that sentinel means "no downbeat has ever locked", and on the way *out* of a silence it would snap 0 → live while `bar_phase` resumed mid-bar, breaking the one-atomic-event invariant that lets A8b share the rate across the thread boundary at all. Both new tests were confirmed to fail against the unfixed code, so they pin the behaviour rather than merely describing it. Verified end-to-end with a synthetic 120 BPM 4/4 signal (accented downbeats, per-bar chord changes) through a PulseAudio null sink, watching OSC across the music→silence transition: under music the detector locked **120.3 BPM** and `bar_phase` swept **128 distinct values per second**; under silence it collapsed to **exactly one value, 0.000**, alongside a pinned `beat_phase` — where before it would have swept on through the quiet. (board #1598)
- **Quitting during an audio stall could hang the process** — `Drop for AudioSystem` dropped `_capture` inline, which joins the capture thread; if that thread was blocked in a timeout-less `pa_simple_read`, the join never returned and the app hung on exit rather than closing. It now hands the backend to the A9 detached reaper, so the process exits and the OS reclaims the parked thread. Pre-existing since the PulseAudio backend landed; found while designing #1460, whose reaper fixes it for free. (board #1460)
- **PulseAudio read errors retried forever at 10 Hz** — the capture loop logged, slept 100 ms and `continue`d on every failed `pa_simple_read`, with no exit path. But `pa_simple` has **no recovery**: once the server kills the stream *every* subsequent read fails, so a dead stream span-locked a thread logging an error ten times a second for the rest of the session. It now gives up after 10 consecutive errors (~1 s), publishes `capture_failed` and exits — which both stops the spam and lets the A9 watchdog reconnect (and makes the reaper's join return immediately instead of parking a thread). A successful read resets the counter, so transient errors are unaffected. (board #1460)
- **Device switch stranded an in-progress recording's audio** — `switch_device` swapped in a fresh `recording_ring`, but a recording started earlier holds a **clone of the old one** (`main.rs`), so its writer thread was left draining a ring nobody writes to and the rest of the take recorded silence. The ring is now threaded through the reopen and deliberately not swapped — the same trick already used for the `tuning` and `tempo` Arcs — so the new audio thread writes to the ring the recording is already reading. This is sound only because the outgoing analysis thread is joined before the new one spawns (`RingBuffer::push` is single-producer). Pre-existing for manual switches; A9's auto-reconnect would have made it fire without the user asking. Note the recording still captures `sample_rate` at start, so a reopen onto a device with a different rate pitch-shifts the remainder — rare, and better than silence. (board #1460)
- **`beat_phase` dropped to 0 several times a beat on loud audio** — the detector's phase-freeze gate tested `rms < 1e-4`, but by the time `BeatDetector::process` receives `rms` it has been through the adaptive normalizer (A2 #1453), which maps it to `(v − P5) / (P95 − P5)` and therefore **floors it at exactly 0.0 whenever the signal touches the bottom of its own recent range**. On any rhythmic material that is the trough between every hit — so the gate fired on *perfectly loud* audio, punching a spurious 1-hop `beat_phase` dropout to 0 and manufacturing false wraps (a 120 BPM signal read **2.12 wraps/sec instead of 2.00**). It now gates on the **A10 perceptual silence flag** (momentary < −55 LUFS) that the detector was already being passed for exactly this purpose (#1457) and that the normalizer itself gates on — an absolute, device-independent measure that a loud trough can't trip. This made the `rms` parameter dead, so it is removed from `process`. Measured on a 120 BPM null-sink signal: spurious zeros **164 → 0**, wraps **2.121 → 1.998/sec** against 2.000 expected. Pre-existing since the A2 normalizer landed, and invisible in the OSC stream because the `rms` broadcast is post-*smoother* (release 0.12 s), which hides the momentary zero. Found while verifying A8 (#1459), whose smooth-phase goal it defeats. (board #1459)
- **BPM test harness fed frames at the wrong clock** — `run_bpm_convergence_test` spaced its synthetic kicks at `dt = 0.01` (100 Hz), but since A5 (#1456) the detector derives frame time from the fixed 512-sample analysis hop (`sr / ANALYSIS_HOP` ≈ 86.1 Hz), so every `target_bpm` in those tests actually reached the estimator **13.9% low** — a nominal 172 arrived as 148, 120 as 103. The ±15% tolerance bands were wide enough to hide it, so the tests passed while asserting something other than what they claimed. The harness now spaces frames at the detector's own clock; all existing convergence tests still pass, and now at their stated tempos. Found while adding A7 tempo-prior tests, whose whole point is which octave the estimator picks. (board #1458)
- **Stale/foreign pipeline cache no longer aborts startup** — a `pipeline_cache.bin` written by a different GPU/driver (GPU swap, hybrid graphics, eGPU, or a driver update) is now discarded and rebuilt instead of fatally failing app init. In wgpu 27, `create_pipeline_cache(fallback: true)` does **not** quietly return an empty cache on a device mismatch — it returns an *invalid* cache object that then makes the first `create_render_pipeline` fail ("Failed to initialize app", black-on-launch). The loaded blob is now validated under a validation error scope; if wgpu rejects it (typically "cache data was created for a different device"), the stale file is deleted and an empty valid cache is created, which the next run repopulates. Verified end-to-end by loading an RTX-written cache under lavapipe (recovers with a warn) and confirming a matching cache still loads clean. (board #1507)
